# Hat Protocol

Every meeting uses six named hats inside a three-pass paired-agent flow.

| Hat | Greek Name | Purpose |
| --- | --- | --- |
| Blue | Logos | Order, synthesis, verification, meeting control |
| White | Episteme | Facts, evidence, knowns, unknowns |
| Red | Pathos | Feelings, stakes, human reactions |
| Black | Krisis | Risk, failure, critique, constraints |
| Yellow | Eudaimonia | Upside, usefulness, flourishing |
| Green | Poiesis | Ideas, making, alternatives |

## Rules

- Blue/Logos starts every meeting as the orchestrator.
- First Blue/Logos creates and announces `.local\meetings\<date-slug>\`.
- First Blue/Logos proposes the pair roster and pass plan, then pauses for user edits.
- The live-agent limit is the orchestrator plus three paired agents.
- Pass 1 paired agents run all six hats internally; only final pair proposals are visible.
- Pass 2 shuffled pairs critique, clarify, and append change proposals.
- Pass 3 is final Blue/Logos synthesis and durable-memory deployment.
- If the user chooses to build, final Blue/Logos keeps pair continuity: the same pairs become implementation or deployment lanes by default.
- The user can interrupt, redirect, or supply evidence between passes.
- Final Blue/Logos is the only durable-memory write point.
- Internal hat checkpoints are saved to Markdown, JSON, and `events.jsonl`.
